# 平台接口与版本规划

## 1. 文档目的

本文件用于约束各子产品之间的协作方式，避免模块边界模糊、重复开发和集成失控。

## 2. 平台公共契约

### 2.1 统一对象模型

所有模块必须围绕以下对象协作：

- `Industry`
- `Company`
- `Product`
- `Region`
- `Event`
- `Relation`

### 2.2 统一 ID 规则

- 上层模块不得自行生成新的业务主键。
- 必须使用数据底座产出的标准 ID。

### 2.3 统一状态规则

- 草稿态
- 已验证
- 已废弃
- 冲突待处理

### 2.4 统一时间规则

- 创建时间
- 更新时间
- 生效时间
- 失效时间

## 3. 子产品接口矩阵

### 3.1 02 -> 03

- 输出标准实体
- 输出标准分类
- 输出原始事件
- 输出地理编码结果

### 3.2 03 -> 04

- 输出区域内企业与产业聚合结果
- 输出企业和产业的关系查询结果

### 3.3 03 -> 05

- 输出企业详情所需图谱关系
- 输出上下游和竞争路径

### 3.4 03 + 02 -> 06

- 输出事件挂接所需实体
- 输出可扩散的关系网络

### 3.5 03 + 06 -> 07

- 输出问答上下文
- 输出事件解释上下文
- 输出结构化关系候选的写回入口

## 4. 推荐版本规划

### V0.1 数据与图谱打底

- 完成公司、产业、地域三大实体标准
- 跑通 AI 基建图谱关系

### V0.2 地图与企业探索 MVP

- 上线地图入口
- 上线企业详情和上下游浏览

### V0.3 事件联动 MVP

- 上线事件脉冲
- 上线基础事件影响展示

### V0.4 AI 解释能力

- 上线产业问答
- 上线热点摘要和事件解释

### V1.0 统一产品闭环

- 完整支持产业、地图、企业、事件、AI 的统一探索体验

## 5. 团队协作建议

1. 一个子产品一个主负责人。
2. 公共数据 schema 评审先于功能开发。
3. 跨模块变更必须先改接口契约文档。
4. AI、图谱、地图不得各自维护独立实体定义。

## 6. 最低验收口径

如果满足以下四点，说明拆分是有效的：

1. 任一模块都能说明自己“输入什么，输出什么，不负责什么”。
2. MVP 主链路不依赖单个超级模块硬编码完成。
3. 数据、关系、地图、事件、AI 之间能通过接口衔接。
4. 后续扩展新产业时，主要是补数据，而不是重做系统结构。
