# 总体产品架构

## 1. 目标

将 Industra 从“单一大而全产品”拆分为多个边界清晰的子产品，使团队可以围绕能力域并行建设，并最终汇合为统一的产业认知系统。

## 2. 总体拆分思路

项目不按“前端、后端、AI、数据”这类技术职能拆分，而按以下产品能力域拆分：

1. 数据底座与采集平台
2. 产业知识图谱与关系引擎
3. 地理地图与区域产业视图
4. 企业情报与产业链探索台
5. 事件脉冲与产业传导系统
6. AI 产业分析与知识构建助手
7. 平台接口与版本规划

这样拆分的原因：

- 每个子产品都能独立定义用户价值。
- 每个子产品都能有独立数据输入和接口输出。
- 便于控制复杂度，避免所有需求堆在一个“超级模块”里。
- 可以支持 MVP 先跑通，再逐步增强。

## 3. 统一产品主链路

Industra 的核心用户链路应保持统一：

```text
选择产业
-> 进入区域/地图视图
-> 查看企业分布
-> 打开企业详情
-> 查看上下游与竞争关系
-> 叠加事件影响
-> 获得 AI 解释与总结
```

子产品拆分不能破坏这条主链路，只能在链路节点上承担独立职责。

## 4. 子产品边界

### 4.1 数据底座与采集平台

- 负责原始数据接入、清洗、标准化、更新和质量治理。
- 不直接负责最终用户交互页面。

### 4.2 产业知识图谱与关系引擎

- 负责统一实体模型与关系模型。
- 负责图谱查询和关系推演基础能力。
- 不负责地图渲染和复杂前端交互。

### 4.3 地理地图与区域产业视图

- 负责地图浏览、区域下钻、空间分析和地图可视化表达。
- 不直接定义企业关系逻辑。

### 4.4 企业情报与产业链探索台

- 负责公司详情、产业位置、上下游、竞争、列表对比和探索路径。
- 不负责底层图谱存储和数据采集。

### 4.5 事件脉冲与产业传导系统

- 负责事件流接入、事件识别、影响挂接和扩散路径展示。
- 不负责基础企业画像维护。

### 4.6 AI 产业分析与知识构建助手

- 负责问答、总结、抽取、辅助建图和解释生成。
- 不负责充当唯一事实源，必须基于结构化知识输出。

### 4.7 平台接口与版本规划

- 负责定义模块契约、公共接口、阶段性目标和跨模块验收。
- 不负责具体业务能力实现。

## 5. 公共设计原则

1. 一个实体只能有一个主键体系，不能在多个子产品中重复定义。
2. 一个关系只能由一个主责任模块维护，其他模块通过接口消费。
3. 所有关系和事件都尽量带时间维度。
4. 用户看到的是统一产品，但研发上必须能拆分部署和独立迭代。
5. AI 输出必须可追溯到来源、置信度和审核状态。

## 6. MVP 范围控制

MVP 只聚焦一条最短闭环：

- 产业范围：AI 基建
- 地理范围：中国省市级
- 企业范围：100-200 家核心企业
- 事件范围：少量高价值新闻/公告
- AI 范围：解释和摘要优先，不追求全自动建图

## 7. 阶段性交付要求

### Phase 0

- 完成数据模型、实体标准、首批产业链梳理方法。

### Phase 1

- 跑通 `产业 -> 地图 -> 企业 -> 上下游`。

### Phase 2

- 加入热力图、下钻、搜索、回溯和更多产业。

### Phase 3

- 加入事件传导和 AI 增强。

### Phase 4

- 扩大行业覆盖，形成全产业认知底座。
